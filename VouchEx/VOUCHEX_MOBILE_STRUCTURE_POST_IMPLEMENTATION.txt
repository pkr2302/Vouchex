================================================================================
PART A — TECHNICAL ARCHITECTURE (HOW MOBILE IS ISOLATED FROM DESKTOP)
================================================================================

Stack: React 19, Vite, single App.css + separate mobile.css, Laravel 11 API.
No native app. Mobile browser only (Chrome/Safari on phone).

PRIMARY MOBILE BREAKPOINT: max-width 1024px
  - All premium mobile components use class "mobile-only" (hidden on desktop)
  - Registry tables use class "desktop-only" on .premium-table-wrapper (hidden on mobile where cards exist)
  - mobile.css is imported in main.jsx AFTER App.css
  - App.css ALSO has @media (max-width: 1024px) rules for sidebar fix (older block)

SECONDARY BREAKPOINTS (still active on mobile):
  - 768px: financial period bar stacks, form grids → 1 column, header clock hidden
  - 640px: safe-area insets, dashboard 2-col quick actions, modals full width

DESKTOP GUARANTEE:
  - Above 1025px: .mobile-only { display: none !important }
  - Bottom nav, FAB, mobile search, notifications sheet, registry cards — invisible
  - Desktop sidebar sticky 280px, hamburger hidden, full tables unchanged

Z-INDEX STACK (mobile):
  1200 — Global search overlay, Notifications panel, Reports sheet
  1152 — Filter sheet header (when open)
  1151 — Filter sheet body (when open)
  1150 — Filter backdrop
  1101 — FAB button
  1100 — Bottom navigation bar
  1099 — FAB backdrop
  1000 — Sidebar drawer (when open via Menu)
  999  — Sidebar backdrop
  50   — Sticky header

B5. GLOBAL SEARCH (MobileGlobalSearch)
--------------------------------------------------------------------------------

Trigger: Bottom nav "Search" tab
Layout: Full-screen overlay, z-index 1200, bg var(--bg-primary)

HEADER:
  Padding: max(12px, safe-area-top) 12px 12px
  Flex row: [search input wrap flex 1] [close 44×44px]
  Input wrap: height 48px, border-radius 12px, padding 0 14px, font 16px (no iOS zoom)
  Placeholder: "Customers, invoices, GSTIN, vouchers…"
  Close: X icon 22px

BODY: scrollable, padding 16px

SECTION "Recent" (when query empty AND localStorage has recent items):
  Up to 8 items from localStorage key vouchex.mobile.recent
  Max 12 stored total

SECTION "Results" (when query non-empty):
  Client-side index built from in-memory data (no API call)
  Max 30 results, multi-term AND match on label+sublabel+keywords

SEARCH INDEX INCLUDES:
  ✓ Customers → tab customer-master (name, GSTIN, email, phone, PAN)
  ✓ Vendors → tab vendor-master
  ✓ Invoices → tab sales
  ✓ Expenses → tab expense
  ✓ Purchases (expenses with record_type purchase) → tab purchase
  ✓ Receipts → tab receipt
  ✓ Payments → tab payment
  ✓ Inventory products → tab inventory
  ✓ Company GSTIN entry → tab settings

SEARCH INDEX DOES NOT INCLUDE:
  ✗ GL ledger accounts / ledgers by name
  ✗ Credit notes, debit notes individually
  ✗ Journal numbers / day book lines
  ✗ Direct deep-link to specific record (opens tab only, not record detail)

Result row layout:
  Grid 3-col: [vertical type label] [title + subtitle] [arrow]
  Padding 14px, border-radius 12px, margin-bottom 8px
  Type label: 10px uppercase vertical blue text

PART G — REASONS FOR NON-IMPLEMENTATION (FOR CHATGPT CONTEXT)
================================================================================

1. REGISTRY CARDS NOT EVERYWHERE
   Reason: Scope/time — each registry needs custom card renderCard JSX. Only highest-traffic registries done first (sales, customers, expenses, inventory).
   Alternative: premium-table-wrapper horizontal scroll + touch scrolling.

2. GLOBAL SEARCH NO LEDGERS / NO DEEP LINKS
   Reason: Ledger accounts require API call (glLedgerAccounts); search is client-side in-memory only for speed/zero backend changes. Deep-link would need URL routing or tab state params not in current SPA architecture (tabs are useState only, no URL paths).

3. FAB DOES NOT OPEN ADD FORM DIRECTLY
   Reason: Each tab manages its own showAddForm state internally in App.jsx subcomponents; no global form-open API. FAB only calls navigateTab(tabId).

4. NO SWIPE GESTURES
   Reason: Zero paid libraries rule — native touch handlers would need careful conflict with scroll. Deferred.

5. NO SERVICE WORKER / FULL PWA
   Reason: Risk to cPanel deployment and cache invalidation on accounting app; manifest-only prep for future Objective 3 (native/PWA app).

6. COMPANY 360 CHARTS NOT REDESIGNED
   Reason: Charts use existing Recharts/draggable grid; mobile CSS stacking deemed sufficient for v1.

7. DEBTORS/CREDITORS/CASH-BANK NO SUBTAB MOBILE MENU
   Reason: Only 2 sub-tabs each — horizontal scroll still usable; MobileSubTabMenu wired only where 4–6 sub-tabs (Receipts, Payments, Taxation, Settings).

8. COA / FINANCIALS SUBTABS STILL HORIZONTAL
   Reason: Financial subtabs explicitly exempted in mobile.css (.financial-subtabs { display:flex }) because vertical menu + horizontal pills was judged OK for 4 TB/P&L/BS/Notes switches.

9. TAXATION CONTENT STILL TABLE-HEAVY
   Reason: GSTR tables have 10+ columns; hiding GSTIN/tax cols in registers but not in taxation tab. Major effort to card-ify GSTR rows.

10. DESKTOP 100% UNCHANGED RULE
    Reason: All mobile features gated behind max-width 1024px and .mobile-only/.desktop-only classes — constrained some shared-component refactors.
